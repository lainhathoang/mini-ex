JWT Auth Implementation Guide

Target: /Users/lainhathoang/Desktop/mini-ex/crates/http-server

This document explains the simplest way to implement username/password authentication with JWT in crates/http-server, based on the current code layout and database schema.

1. Current State

  • The database already has a users table.
  • The table already includes username and password_hash.
  • The auth router in src/handlers/auth/mod.rs is still unimplemented.
  • The JWT extractor still assumes wallet-style claims using address.
  • The JWT signing helper exists conceptually but is commented out.
Important: the database model is already username/password based, but the HTTP auth layer is still wallet-oriented. The first job is to align the JWT claim shape with the user model.

2. Minimal Design

Keep the first version small: sign up, sign in, JWT generation, JWT validation, and one protected route.

JWT claim shape

  • sub: user id as the main identity field
  • username: optional convenience field
  • exp: expiration timestamp

Why this shape

  • sub is the stable identity key.
  • username can change later, so it should not be the primary identifier.
  • exp is already compatible with your current JWT validation flow.

3. Endpoints to Implement

Public endpoints

  • POST /auth/sign-up
  • POST /auth/sign-in

Protected endpoint

  • GET /users/me

Optional later endpoint

  • POST /auth/sign-out
Recommendation: keep transport as Authorization: Bearer <token> for the first implementation. That matches the current extractor structure and is the least disruptive change.

4. Endpoint Responsibilities

POST /auth/sign-up

  • Accept username and password.
  • Trim and validate input.
  • Reject duplicate usernames.
  • Hash the password.
  • Create the user record.
  • Return either a success message or an auth token.

POST /auth/sign-in

  • Accept username and password.
  • Find the user by username.
  • Verify the provided password against password_hash.
  • Create a JWT using the user id.
  • Return { "token": "..." }.

GET /users/me

  • Read the bearer token.
  • Decode and validate the JWT.
  • Extract the user id from sub.
  • Load the user from the database.
  • Return current user data.

5. Files to Change

src/handlers/auth/mod.rs

  • Replace todo!() with a router containing sign-up and sign-in routes.
  • Define request payload structs for sign-up and sign-in.
  • Keep the handlers focused on validation, repository calls, password hashing, and token generation.

src/extractors/auth.rs

  • Replace the wallet-specific claim shape.
  • Use claims like sub, username, and exp.
  • Keep the extractor behavior the same: read bearer header, decode JWT, reject invalid tokens.

src/common/jwt.rs

  • Implement the signing helper.
  • Use Env::AccessTokenKey as the signing secret.
  • Set a simple expiration time such as 1 day or 3 days.

src/handlers/users/me.rs

  • Stop assuming an authenticated wallet address.
  • Use the authenticated user id from JWT claims.
  • Load the user from the database and return a user-shaped response.

Database repository layer

  • Add a query to find a user by username.
  • Add a query to find a user by id.
  • Add a create-user operation.

6. Request and Response Shapes

Sign-up request

{
  "username": "alice",
  "password": "your-plain-password"
}

Sign-in request

{
  "username": "alice",
  "password": "your-plain-password"
}

Sign-in response

{
  "token": "jwt-token-here"
}

Protected request header

Authorization: Bearer <jwt-token>

/users/me response

{
  "id": "user-uuid",
  "username": "alice",
  "cash_balance": "0"
}

7. Validation Rules

  • Username length: 3 to 32 characters.
  • Password length: at least 8 characters.
  • Reject empty strings after trimming.
  • Reject duplicate usernames at sign-up.
  • Return a generic sign-in error like Invalid credentials.

8. Password Hashing Rules

  • Never store the raw password.
  • Store only password_hash.
  • Use Argon2 or bcrypt.
  • Argon2 is the better default for new work.
Do not use: MD5, SHA1, SHA256, or any fast general-purpose hash as a password storage algorithm.

9. JWT Rules

  • Use HS256 for the simplest setup.
  • Use the existing AccessTokenKey secret.
  • Set an expiration time and reject expired tokens.
  • Reject malformed and invalid tokens as 401 Unauthorized.

10. Suggested Implementation Order

  1. Define the new JWT claim structure.
  2. Implement JWT signing in src/common/jwt.rs.
  3. Add repository methods for user lookup and user creation.
  4. Implement POST /auth/sign-up.
  5. Implement POST /auth/sign-in.
  6. Update the auth extractor to decode the new claims.
  7. Update /users/me to load by authenticated user id.
  8. Update the OpenAPI document after the handlers work.

11. Error Model

  • 400 Bad Request: invalid input, username already taken
  • 401 Unauthorized: bad credentials, missing token, invalid token, expired token
  • 500 Internal Server Error: database failure, hashing failure, signing failure

12. First-Version Scope

Keep version one limited to:

  • Sign up
  • Sign in
  • JWT generation
  • JWT validation
  • One protected route

Do not add these yet unless they are clearly needed:

  • Refresh tokens
  • Password reset
  • Email verification
  • Role-based authorization
  • Multi-session device management

13. Done Criteria

  • A new user can register with a unique username.
  • An existing user can sign in with the correct password.
  • Sign-in returns a JWT.
  • /users/me only works with a valid token.
  • Expired and invalid tokens return 401.
